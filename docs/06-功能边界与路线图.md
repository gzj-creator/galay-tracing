# 功能边界与路线图

本文档说明当前 `galay-tracing` 已支持什么、还缺什么，以及建议的补齐顺序。

## 已支持

- 固定二进制 `TraceId`、`SpanId`，hex parse/format，非零校验。
- W3C `traceparent` extract/inject。
- Opaque `tracestate` 保存和透传。
- `TraceContext`、`SpanContext`、`LogContext` 分层。
- `startSpan()`、`startServerSpan()`、嵌套 span 和作用域恢复。
- Span kind、status 和 bounded owning attributes。
- Always on/off、parent-based 和 trace-id-ratio sampler。
- 可选 span timestamp，默认关闭。
- Exporter-level resource attributes 和 instrumentation scope。
- 日志关联：普通日志、context-bound logger、结构化 event、宏快路径。
- concept-based custom writer，不要求继承虚基类。
- `SpdlogSink`。
- `galay-kernel` 显式 context capture/logging。
- `BatchSpanProcessor`、`FileSpanExporter`、OTLP/HTTP JSON exporter。
- 构建、CTest、examples、benchmark 和 Rust tracing 对照 benchmark。

## 重要边界

当前 span 仍是轻量生命周期对象，不是完整 OpenTelemetry Span 语义模型。它有 trace/span identity、name、parent、sampled flag、tracestate、kind、status、bounded attributes 和可选 timing，但还没有 events/links 等完整字段。Resource 和 instrumentation scope 作为 exporter 配置输出，不放进每个 span 对象。

当前 exporter 是 HTTP JSON 路径，适合接入支持 OTLP/HTTP JSON 的 collector 或内部服务。需要标准 OTLP/protobuf 或 gRPC 时，要新增 exporter。

当前上下文传播在同步代码中由 TLS 和 `SpanGuard` 处理；协程中推荐显式捕获和传递 context。库没有自动 task resume hook。

## 缺口优先级

1. Span 事件和链接：
   - bounded events：name + timestamp + attributes。
   - links：remote context + attributes。

2. HTTP/RPC middleware：
   - inbound extract helper。
   - outbound inject helper。
   - 基于 generic header get/set concept，避免强绑某个 HTTP 类型。

3. Exporter 能力：
   - retry/backoff。
   - compression。
   - OTLP/protobuf HTTP。
   - OTLP/gRPC。
   - HTTPS transport。

4. Baggage：
   - header parse/inject。
   - 明确大小限制和复制策略。

## 不建议马上做

- 全局隐式协程 resume hook：侵入大，和 `galay-kernel` 调度语义耦合，需要单独设计。
- 过早引入 protobuf/gRPC 依赖：会放大包体和构建复杂度。
- 把日志 writer 改回虚基类：当前 concept + erased snapshot 对热路径更友好。
