# 性能测试

本页记录 `galay-tracing` 自带 benchmark 的用途和运行方式。benchmark 是独立可执行程序，不作为 CTest 通过条件的一部分。

## 构建

```bash
cmake -S . -B build-release -DCMAKE_BUILD_TYPE=Release -DBUILD_TESTING=ON -DBUILD_EXAMPLES=ON -DBUILD_BENCHMARKS=ON
cmake --build build-release --parallel
```

## 程序

- `B1-DisabledLog`：禁用日志级别时的热路径开销，预期不写 sink。
- `B2-EnabledLog`：启用日志并写入空 sink 时的格式化与分发开销。
- `B3-SpanScope`：sampled 与 unsampled 子 span 作用域创建和恢复开销。span timestamp 默认关闭、可通过配置开启；同时打印 span/trace ID 生成成本和 subscriber-like noop span scope 作为 Rust `tracing` noop 对照基线。
- `B4-Traceparent`：W3C `traceparent` parse/inject 往返开销。
- `B5-OtlpJsonExporter`：OTLP/HTTP JSON 批量编码到 mock transport 的每 span 开销。
- `B6-StructuredEvent`：结构化事件 disabled 与 enabled no-op writer 热路径开销，覆盖显式 writer、默认 writer、Rust-like callsite 宏路径、默认 writer 宏快路径，以及降级到 `Logger` sink 的 fallback 路径。
- `B7-BatchProcessorSchedule`：对比 `BatchSpanProcessor` 的 `kTimed`、`kOnEnd`、`kBatchSize` 三种后台线程唤醒策略。每种策略运行多轮并打印 best/median 的 `onEnd`、flush 和端到端每 span 开销。
- `B8-MoodycamelBatchQueue`：隔离测试 `moodycamel::ConcurrentQueue` + batch-size 唤醒 + bulk drain 的队列层成本，用于和完整 `BatchSpanProcessor` 成本拆分对照。

每个程序都会打印 workload、build type、backend/adapters 和关键 ns/op 指标，便于后续 release note 引用。

## Rust 对比

`benchmark/compare/rust-tracing` 是独立 Cargo 基准，用于在同一机器上对比 Rust `tracing` 的 disabled/enabled no-op/span scope、`serde_json` 生成同形 OTLP JSON 的开销，以及两组 span-like owned payload 批处理基线：

- `R5-RustCrossbeamBatchProcessor`：`crossbeam-channel` bounded queue + batch worker，作为 Rust 开源库完整批处理主对照。
- `R6-RustCrossbeamArrayQueueBatchSize`：`crossbeam-queue::ArrayQueue` + batch-size unpark，作为更贴近 C++ `kBatchSize` 唤醒策略的队列层参考。

运行：

```bash
cargo run --release --manifest-path benchmark/compare/rust-tracing/Cargo.toml
```

做 batch processor 对比时优先比较 `B7-BatchProcessorSchedule mode=batch_size` 和 `R5-RustCrossbeamBatchProcessor`：两者都是 100000 条 span-like owned payload、batch size 512、bounded queue、后台线程批量消费、显式 flush、7 轮 best/median 统计。payload name 固定为 30 字节，避免 C++ `std::string` 小字符串优化和 Rust `String` 指针移动差异影响结论。`B8-MoodycamelBatchQueue` 和 `R6-RustCrossbeamArrayQueueBatchSize` 用于解释队列与唤醒策略本身的成本，不替代完整 processor 对比。Rust `tracing` 的 no-op subscriber 指标只用于框架 callsite 热路径参考，不等价于完整异步 batch processor。

本机最近一次 Release 参考结果：

| 项目 | 口径 | median e2e |
| --- | --- | ---: |
| `B7 mode=batch_size` | 完整 `BatchSpanProcessor`，moodycamel queue，按 batch size 唤醒 | 14.23 ns/span |
| `R5-RustCrossbeamBatchProcessor` | `crossbeam-channel` bounded queue + batch worker | 27.54 ns/span |
| `B8-MoodycamelBatchQueue` | 仅 C++ 队列层和 batch-size 唤醒 | 13.72 ns/span |
| `R6-RustCrossbeamArrayQueueBatchSize` | Rust `ArrayQueue` 队列层和 batch-size unpark | 43.65 ns/span |

基准数据受 CPU 频率、系统负载和调度影响，引用时应重新在目标机器运行同一组命令。
